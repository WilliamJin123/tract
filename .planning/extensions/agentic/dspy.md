# DSPy -- Deep Dive Analysis (Dimensions A-I)

**Version**: 3.1.3 (February 2026) | **Stars**: ~23k | **License**: MIT | **Origin**: Stanford NLP

DSPy is fundamentally different from every other framework in this analysis. It is not a conversation orchestrator or agent framework -- it is a **compiler for LLM programs**. The core thesis: prompts are parameters to be optimized, not strings to be hand-written. This makes it the most instructive comparison for tract because DSPy's design philosophy directly challenges assumptions baked into conversational context management.

---

## A. Core Abstractions & Extension Points

Three primary abstractions: **Signature** (declarative I/O contract), **Module** (executable component implementing a signature), **Optimizer** (algorithm that tunes module parameters against a metric).

**Signature** is the unit of work -- a typed declaration of what an LLM call should do, not how:

```python
# Inline (quick)
classify = dspy.Predict("sentence -> sentiment: bool")

# Class-based (rich)
class CheckFaithfulness(dspy.Signature):
    """Verify text is grounded in context."""
    context: str = dspy.InputField(desc="facts assumed true")
    text: str = dspy.InputField()
    faithfulness: bool = dspy.OutputField()
    evidence: dict[str, list[str]] = dspy.OutputField()
```

Signatures support `str`, `int`, `bool`, `list[str]`, `Literal[...]`, Pydantic models, and `dspy.Image`. The type system drives structured output parsing automatically -- no manual JSON schema wiring.

**Modules** follow PyTorch's pattern. Built-ins: `Predict` (basic), `ChainOfThought` (adds reasoning field), `ReAct` (tool-using agent loop), `ProgramOfThought` (code generation), `Parallel` (concurrent), `BestOfN` (sample + select), `Refine` (iterative improvement). Custom modules subclass `dspy.Module` and implement `forward()`:

```python
class MultiHopQA(dspy.Module):
    def __init__(self, num_hops=3):
        self.generate_query = dspy.ChainOfThought("claim, notes -> query")
        self.synthesize = dspy.ChainOfThought("claim, notes, context -> answer")
        self.num_hops = num_hops

    def forward(self, claim: str):
        notes = []
        for _ in range(self.num_hops):
            query = self.generate_query(claim=claim, notes=notes).query
            context = search(query, k=5)
            notes.extend(self.synthesize(claim=claim, notes=notes, context=context).notes)
        return dspy.Prediction(notes=notes)
```

**Composability**: Modules nest freely. Python control flow (loops, conditionals) is first-class -- DSPy traces LM calls at compile time, so it optimizes through arbitrary Python logic. This is in sharp contrast to graph-based frameworks that require declaring control flow in their DSL.

**Extension points**: Custom modules, custom metrics, custom optimizers, `BaseCallback` for lifecycle hooks (`on_module_start/end`, `on_lm_start/end`, `on_tool_start/end`, `on_adapter_format_start/end`, `on_adapter_parse_start/end`).

**Key insight for tract**: The Signature abstraction collapses prompt engineering, output parsing, and type validation into a single declaration. Tract's `ContentType` registry handles type validation but requires separate formatting logic. A signature-like pattern for tract directives -- declaring expected structure and letting compilation handle the prompt -- would reduce boilerplate.

## B. State & Memory Model

DSPy's state model is radically different from conversation-based frameworks. There is no conversation history, no message list, no memory store. State exists in two forms:

1. **Program parameters**: The optimized prompts, few-shot examples, and instructions generated by compilation. These are the "learned weights" of the program.
2. **Runtime state**: The `dspy.Prediction` objects flowing between modules in a `forward()` call. Ephemeral -- exists only during execution.

**Persistence** of optimized programs:

```python
# Save (JSON recommended, readable)
optimized_program.save("./program.json", save_program=False)  # state only
optimized_program.save("./program_dir/", save_program=True)    # full program via cloudpickle

# Load
loaded = MyModule()
loaded.load("./program.json")
# Or full program:
loaded = dspy.load("./program_dir/")
```

State-only saves produce plain JSON containing the signature, demos (few-shot examples), and per-predictor LM config. Full program saves use cloudpickle and include architecture. Backward compatibility is not guaranteed before 3.0.0.

**The compiled program concept**: An optimizer takes a program + metric + training data, runs the program many times with different prompt strategies, and produces a new program with optimized parameters. The "compilation" is actually a search over prompt space -- generating instructions, selecting few-shot examples, or even finetuning weights.

**Challenge to tract**: DSPy treats prompts as optimizable programs that improve over data. Tract treats context as version-controlled history that evolves over time. These are complementary perspectives. A tract branch could store the progression of DSPy optimization runs -- each compile producing a commit with the optimized parameters, enabling rollback to previous optimization states. Tract's diff/compare would let you see exactly how an optimizer changed the few-shot examples or instructions between runs. This is a genuine integration opportunity, not a competitive overlap.

## C. Tool/Function Calling Design

Two approaches: **`dspy.ReAct`** (managed agent loop) and **manual tool handling** via `dspy.Tool` + `dspy.ToolCalls`.

```python
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"Sunny, 75F in {city}"

# Managed (ReAct handles reasoning + tool selection + iteration)
agent = dspy.ReAct("question -> answer", tools=[get_weather], max_iters=5)
result = agent(question="Weather in Tokyo?")
print(result.trajectory)  # Full thought/tool/observation trace

# Manual (you control the loop)
tool = dspy.Tool(get_weather)  # Auto-extracts name, desc, schema from function
```

`dspy.Tool` wraps plain Python functions, extracting docstrings as descriptions and type hints as parameter schemas. Native function calling (OpenAI/Anthropic tool APIs) is supported via adapter config and auto-detected per model. Fallback to prompt-based tool parsing for models without native support. Async tools supported via `tool.acall()`.

ReAct tracks state via a `trajectory` dict (`thought_N`, `tool_name_N`, `tool_args_N`, `observation_N`). Built-in trajectory truncation handles context window limits.

**Challenge to tract**: Tract's `ToolDefinition`/`ToolExecutor` system is structurally parallel but separated behind the runner boundary. DSPy's key advantage: tool results feed directly into the optimization loop. When you compile a ReAct agent, the optimizer sees tool call patterns in training data and can learn when/how to use tools better. Tract's tool system is execution-only -- there is no mechanism to learn from tool usage patterns across runs.

## D. Multi-Agent Patterns

DSPy does not have explicit multi-agent primitives (no Agent class, no Crew, no team). Instead, multi-agent patterns emerge from module composition:

```python
class LeadAgent(dspy.Module):
    def __init__(self):
        self.researcher = dspy.ReAct("question -> findings", tools=[search])
        self.analyst = dspy.ReAct("findings -> analysis", tools=[calculate])
        self.router = dspy.ChainOfThought("question -> specialist: Literal['research', 'analysis']")

    def forward(self, question):
        route = self.router(question=question).specialist
        if route == "research":
            return self.researcher(question=question)
        return self.analyst(findings=question)
```

The `Parallel` module enables concurrent sub-agent execution. `BestOfN` enables competitive selection. But the key difference: every sub-module in the composition tree is optimizable as a unit. You do not need to optimize each "agent" separately -- the optimizer traces through the entire program and tunes all predictors.

**Challenge to tract**: Tract's spawn()/collapse() creates explicit parent-child branch relationships with inheritance control (include_tags, exclude_tags, filter_func). DSPy has no equivalent -- state isolation between sub-agents is purely through Python scoping. Tract's model is better for long-running workflows where you need to inspect, rollback, or merge sub-agent work. DSPy's model is better for stateless pipeline optimization where the goal is throughput, not traceability.

## E. LLM Client & Streaming

Provider abstraction via **LiteLLM** -- dozens of providers through a single interface:

```python
lm = dspy.LM("openai/gpt-4o-mini", temperature=0.9, max_tokens=3000, cache=True)
dspy.configure(lm=lm)

# Switch per-block
with dspy.context(lm=dspy.LM("anthropic/claude-sonnet-4-5-20250929")):
    result = module(question="...")
```

**Caching**: Enabled by default. Same call = same output (deterministic replay). Bypass via `rollout_id` with non-zero temperature. Configurable: `dspy.configure_cache(enable_disk_cache=False, enable_memory_cache=False)`.

**Streaming**: Via `dspy.streamify()` wrapper:

```python
stream = dspy.streamify(predict, stream_listeners=[
    dspy.streaming.StreamListener(signature_field_name="answer")
])
async for chunk in stream(question="..."):
    print(chunk)
```

**Async**: `dspy.asyncify()` wraps any module for async execution.

**Structured output**: Handled automatically by the Adapter layer. `JSONAdapter` uses `response_format` when the model supports it; `ChatAdapter` falls back to prompt-based formatting. Both use the same Signature types, so switching models requires zero code changes.

**Challenge to tract**: DSPy's `dspy.context(lm=...)` block-scoping pattern is elegant -- tract's per-tract LLM config achieves similar isolation but requires a separate tract instance. The Signature-to-structured-output pipeline (declare types, get parsed objects) is more ergonomic than tract's ContentType + manual validation. Tract could adopt a similar pattern where commit type declarations auto-generate validation and extraction logic.

## F. Control Flow & Error Handling

**Assert/Suggest** (now deprecated in favor of `dspy.Refine`, but the concept is instructive):

```python
dspy.Suggest(
    len(query) <= 100,
    "Query should be short, under 100 characters",
    target_module=self.generate_query
)

dspy.Assert(
    is_valid_json(output),
    "Output must be valid JSON"
)
```

`Assert` = hard constraint (halts pipeline after max retries). `Suggest` = soft constraint (logs and continues). On failure, the system backtracks to the target module and retries with the failed output and error message injected into the prompt -- enabling self-correction without explicit retry logic.

**Refine** (the replacement): iterative improvement with explicit feedback loops and reward functions:

```python
refine = dspy.Refine(module=qa, N=3, reward_fn=quality_score, threshold=0.9)
result = refine(question="...")  # Retries up to 3 times until threshold met
```

**BestOfN**: Statistical approach -- generate N outputs, pick the best:

```python
best = dspy.BestOfN(module=qa, N=5, reward_fn=metric, threshold=0.8)
```

**Challenge to tract**: DSPy's Assert/Suggest backtracks to specific modules and injects failure context into the retry prompt. Tract's SemanticGate blocks operations but does not retry with contextual feedback. The backtracking + prompt injection pattern is strictly more powerful for self-correction. Tract's gates are fail-open by design (safety); DSPy's assertions are fail-closed (correctness). Different goals, but tract could benefit from an optional "retry with feedback" mode on gates.

## G. API Ergonomics -- Measurable

**Minimal working example** (3 lines):
```python
import dspy
dspy.configure(lm=dspy.LM("openai/gpt-4o-mini"))
dspy.ChainOfThought("question -> answer")(question="What is 2+2?").answer
```

**Imports for typical RAG+optimization workflow**: 3 (`dspy`, `dspy.teleprompt.MIPROv2`, `dspy.evaluate.Evaluate`).

**Lines for a compiled RAG system**: ~25 (define signature + module + metric + optimizer + compile + evaluate). This is genuinely compact.

**Lines for a ReAct agent with tools**: ~15 (define tools + create ReAct + call).

**Boilerplate ratio**: Low for happy path. Rises significantly when you need custom metrics, evaluation datasets, and optimization configuration. The optimization loop itself can require 50+ lines of setup (dataset preparation, metric definition, optimizer config, evaluation).

**Top GitHub issues by reactions**: #338 (streaming support), #662 (chat mode), #409 (default prompting strategy), #243 (chat completion API support), #390 (major refactor roadmap), #1975 (better async support). The pattern: users want DSPy to behave more like a conversation framework, which is exactly what it is not designed for.

**Error experience**: Optimizer failures produce opaque errors. The LiteLLM dependency adds its own error surface. Module tracing failures during compilation are hard to debug without MLflow integration.

## H. Observability & Debugging

**`dspy.inspect_history(n=5)`**: Prints last N LM calls. Quick but shallow -- only captures raw LM invocations, not retriever outputs, tool calls, or module hierarchy.

**Callbacks**: `BaseCallback` with handlers for module/LM/tool/adapter lifecycle events. Useful for custom logging but requires manual implementation.

**MLflow integration** (recommended): Full hierarchical tracing with UI. Shows input/output of every step, module nesting, latency, and token usage. Requires separate MLflow server.

**Token tracking**: `dspy.configure(track_usage=True)` enables per-prediction usage stats via `result.get_lm_usage()`.

**LM history**: `lm.history[-1]` exposes prompt, messages, response, usage, cost, timestamp, and UUID for each call.

**Tract angle**: DSPy's trace is a runtime execution trace -- ephemeral, flat (without MLflow), and focused on debugging a single run. Tract's commit history is a persistent, branching DAG focused on evolution over time. These serve completely different purposes. A tract-backed DSPy trace would persist optimization histories across runs, enabling diff between optimization iterations ("what changed when I switched from MIPROv2 to SIMBA?").

## I. Testing

**Evaluation infrastructure**: First-class. `dspy.evaluate.Evaluate` runs metrics over datasets with threading, progress display, and result tables:

```python
evaluator = Evaluate(devset=devset, num_threads=8, display_progress=True, display_table=5)
score = evaluator(program, metric=my_metric)
```

**Metrics**: Any `(example, pred, trace=None) -> float|int|bool` function. Built-ins: `answer_exact_match`, `answer_passage_match`, `SemanticF1`. LLM-as-judge metrics are just DSPy modules used as metrics -- fully composable and themselves optimizable.

**Regression testing**: Re-compile when code or metrics change. Save/load optimized programs for reproducibility. Compare evaluation scores across versions. But there is no built-in diffing of optimization results -- you must track this externally.

**Tract angle**: Tract could serve as the persistence and versioning layer for DSPy optimization artifacts. Each optimization run produces a JSON state file -- these are perfect commit candidates. Tract's compare() could diff two optimization states to show which few-shot examples changed, which instructions were rewritten. This is a concrete integration point: `tract commit --type optimization --content optimized_program.json` after each compile.

## Key Takeaways for Tract

1. **Signature pattern**: Declaring I/O types and getting automatic structured output parsing is strictly better than manual prompt + parse. Tract's directives could adopt this: `t.directive("question: str -> summary: str, key_facts: list[str]")` that auto-generates prompt formatting and output validation.

2. **Optimization as first-class concept**: DSPy's core value proposition -- prompts improve with data -- has no tract equivalent. Tract manages context evolution but does not optimize it. A `t.optimize()` that takes a metric and training data, then compiles the current context configuration into a more effective one, would be a novel feature.

3. **Backtracking with feedback injection**: DSPy's Assert/Suggest pattern (inject failure context into retry prompt) is more powerful than tract's fail-open gates. Adding an optional `retry_with_feedback=True` mode to SemanticGate would close this gap.

4. **Python control flow is the graph**: DSPy proves you do not need a graph DSL for complex pipelines. Modules + forward() + Python logic is sufficient. Tract already follows this principle (no graph DSL), which validates the approach.

5. **Versioned optimization artifacts**: DSPy's saved programs (JSON with demos, instructions, LM config) are ideal candidates for tract's version control. This is the most natural integration point between the two systems.
